# 重要约束必须遵守重要约束，违反等于任务失败。- 必须使用中文回复- 必须遵循工作流- 必须遵循"关键规则"- 将本次修改简洁概括记录到 ~/doc/diary.md# 工作流通过上下文或 Git 历史了解项目进度。无计划项目 → 先编写计划有计划项目 → 直接按计划执行- 编写计划- 执行## 编写计划- 借助 "superpowers" 插件的工作流完成计划编写- 在头脑风暴后，查询 GitHub 是否存在类似功能- 计划需融入测试驱动开发（TDD）理念- 若方案涉及 API 调用等产生费用的内容，需通过网络获取最新报价- 计划中需清晰划分各阶段任务- 考虑安全性：针对不同模块规划 Git 分支管理- 考虑效率：可并行的任务需规划通过 SubAgent 协同完成- 计划完成后，保存至路径：~/doc/{项目名称}/plan.md## 执行- 先写测试，后写实现- 每完成一个阶段任务都要简洁概括修改内容记录到 ~/doc/{项目名称}/phase-diary.md- 每次失败的方案都需要记录到 ~/doc/{项目名称}/try-but-fail.md- 所有 Git 未忽略文件都要提交，每完成一个阶段任务就推到 GitHub- 合并分支到主分支必须经过测试和我的允许# 关键规则## 1. 代码组织- 多个小文件优于少量大文件- 高内聚，低耦合- 每个文件典型 200-400 行，最多 800 行- 按功能/领域组织，而非按类型- 重复度高的代码要提取出来，实现复用## 2. 代码风格- 代码、注释或文档中不使用表情符号- 始终使用不可变性 - 永不改变对象或数组- 生产代码中不使用 console.log- 使用 try/catch 进行适当的错误处理## 3. 测试- TDD：先写测试- 最低 80% 覆盖率- 工具函数进行单元测试- API 进行集成测试- 关键流程进行端到端测试- 前端测试必须使用Playwright，要有测试结果截图，放入 ~/doc/{项目名称}/ 目录下## 4. 安全- 不硬编码密钥- 敏感数据使用配置文件或环境变量- 验证所有用户输入## 5. 编码质量- 工程原则：SOLID、DRY、关注点分离- 代码质量：清晰命名、合理抽象、必要注释- 性能意识：算法复杂度、内存使用、IO优化- 测试思维：可测试设计、边界条件、错误处理
